# Dogpile 0.2 Snow Leopard

Status: Executed  
Date: 2026-04-25  
Release type: hardening-only minor release

## Release Contract

Dogpile 0.2 is a Snow Leopard release. It does not add protocols, public API
surface, event fields, provider adapters, runtime capabilities, or new product
claims. Every change in the milestone must fit one of these categories:

- bug fix
- performance improvement backed by a baseline
- release, packaging, or CI reliability improvement
- test hardening
- documentation truthfulness or clarity fix
- internal code debt reduction that preserves public behavior

If a change needs a public API addition to fix a bug, record the bug, the
minimal contract change, and why an internal-only fix is not sufficient before
implementing it.

## Goals

- Make the existing SDK behavior more predictable for application developers.
- Make package, tarball, browser, and downstream-consumer validation boringly
  repeatable.
- Remove drift between code, tests, README, developer docs, changelog, and
  release guard scripts.
- Establish performance baselines before making performance claims.
- Reduce internal duplication where it makes behavior easier to test and reason
  about.

## Non-Goals

- No new first-party coordination protocols.
- No new provider adapter families.
- No new public package subpaths.
- No new event or trace schema fields unless required to fix a documented bug.
- No paper-parity expansion or new reproduction claims.
- No broad architecture rewrite.

## Initial Planning Findings

The milestone opened with one concrete docs/test/release drift finding:

- `package.json` and `scripts/check-package-identity.mjs` pointed at
  `@dogpile/sdk@0.1.2`.
- README release identity text, pack tarball examples, and package export tests
  still expected `@dogpile/sdk@0.1.1`.
- `pnpm exec vitest run src/tests/package-exports.test.ts` failed three
  assertions on that version identity mismatch.

Dogpile 0.1.2 closes that finding in the current worktree by unifying the
manifest, release guard, README, changelog, package export tests, and pack
metadata assertions. Keep this as the first regression target for 0.2: version
bumps should fail early when only part of the release identity surface is
updated.

## Workstreams

### 1. Release Identity And Docs Drift

Unify version, package identity, changelog, README quickstart, pack filenames,
release guard expectations, and package export tests.

Acceptance criteria:

- `@dogpile/sdk` version identity is consistent across manifest, tests,
  release guard scripts, README, changelog, and pack metadata assertions.
- Version-specific README examples use the same tarball name produced by pack.
- Focused package export tests pass.
- Drift checks fail when future version bumps update only part of the release
  identity surface.

### 2. Public Contract Bug Hunt

Audit the behavior callers directly depend on:

- `Dogpile.pile()`, `run()`, `stream()`, and `createEngine()`
- cancellation and timeout propagation
- stream completion, stream errors, and `StreamHandle.status`
- provider failure normalization
- runtime tool validation and execution results
- replay and replay stream behavior
- protocol termination behavior across `sequential`, `broadcast`, `shared`, and
  `coordinator`

Acceptance criteria:

- Each fixed bug has a regression test or a written reason a regression test is
  not practical.
- Public error codes and result shapes remain stable unless a documented bug
  requires a minimal contract correction.
- No new public API surface is added by default.

### 3. Package And Runtime Reliability

Treat package validation as the release center, not a final checklist.

Acceptance criteria:

- `pnpm run verify` passes.
- `pnpm run pack:check` passes.
- `pnpm run publish:check` passes or is blocked only by intentional external
  publish constraints.
- Browser ESM smoke, source-map guard, declaration-map guard, package artifact
  guard, and packed-tarball consumer smoke all exercise the current release
  identity.
- Package export coverage proves public subpaths import from packed artifacts,
  not local source paths.

### 4. Performance Baseline And Targeted Improvements

Measure before optimizing. Focus only on existing behavior.

Candidate baseline areas:

- protocol loop runtime with deterministic providers
- event construction and trace accumulation
- stream subscriber fanout
- trace serialization and replay
- package verification runtime
- packed consumer smoke runtime

Acceptance criteria:

- At least one repeatable benchmark or timing harness exists before perf claims
  are made.
- Any performance improvement records the baseline, the changed result, and the
  command used to compare them.
- Optimizations preserve existing public behavior and test contracts.

### 5. Code Debt Reduction

Prefer small refactors that make existing behavior easier to prove.

Candidate areas:

- shared config validation helpers
- event and result construction duplication
- protocol execution test fixtures
- provider failure normalization helpers
- release identity constants used by scripts and tests

Acceptance criteria:

- Refactors preserve the public API and exported types.
- Refactors are paired with existing or new tests that prove behavior did not
  move.
- No abstraction is added unless it removes meaningful duplication or reduces
  test fragility.

### 6. Documentation Truth Audit

Make docs describe exactly what the SDK does today.

Audit targets:

- `README.md`
- `docs/developer-usage.md`
- `CHANGELOG.md`
- `examples/README.md`
- benchmark fixture notes and reproduction wording
- package scripts and release verification descriptions

Acceptance criteria:

- Docs do not claim unsupported runtimes, unimplemented protocols, unpublished
  package versions, or unproven paper-reproduction outcomes.
- Public examples compile or are covered by an explicit smoke path.
- Release verification docs match package scripts and CI requirements.

## Suggested Execution Order

1. Fix release identity and version drift.
2. Run focused package/export validation.
3. Run the full local gate and record failures.
4. Bug-hunt public contract behavior with targeted regression tests.
5. Add performance baselines.
6. Apply measured performance and debt reductions.
7. Run docs truth audit.
8. Close with `verify`, `pack:check`, and `publish:check`.

## Release Exit Criteria

- `pnpm run verify` passes.
- `pnpm run pack:check` passes.
- `pnpm run publish:check` passes or has only intentional external publish
  blockers.
- Focused package/export tests pass.
- All release-facing docs use the current package identity.
- No accidental public API additions are present in `package.json` exports or
  root type exports.
- Every bug fix has a regression test or an explicit testability note.
- Every performance claim includes a baseline and comparison command.
- Changelog documents 0.2 as a hardening-only release.
