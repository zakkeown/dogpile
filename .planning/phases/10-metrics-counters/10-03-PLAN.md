**Edit 1 — CHANGELOG.md**: Find the `## [0.5.0] — 2026-05-01` header (or the active unreleased version). Add a new section after the Phase 9 entry (or Phase 8 if Phase 9 is not yet in CHANGELOG):

```markdown
### Added — Metrics / Counters (Phase 10)

- **New subpath: `@dogpile/sdk/runtime/metrics`.** Exports `MetricsHook`, `RunMetricsSnapshot`, `buildRunSnapshot`, and `buildSubRunSnapshot`. Callers import from this subpath for type annotations; the `metricsHook` field on `EngineOptions` / `DogpileOptions` accepts an inline object and requires no explicit import.
- **New optional field: `metricsHook?: MetricsHook` on `EngineOptions` and `DogpileOptions`.** When provided, `onRunComplete` fires after every completed run at all recursion depths, and `onSubRunComplete` fires from the parent coordinator's perspective after every completed sub-run. Both callbacks are fire-and-forget: async hooks never block run completion; throwing hooks route errors to the `logger` error channel.
- **New optional field: `logger?: Logger` on `EngineOptions` and `DogpileOptions`.** Uses the existing `Logger` interface from `@dogpile/sdk/runtime/logger`. When absent, hook errors fall back to `console.error`. Adding `logger` now enables future engine-level debug/info logging without another public-surface change.
- **`MetricsHook` interface.** `{ onRunComplete?: (snapshot: RunMetricsSnapshot) => void | Promise<void>; onSubRunComplete?: (snapshot: RunMetricsSnapshot) => void | Promise<void> }`. Both callbacks optional; callers opt into run-level metrics, sub-run-level metrics, or both independently.
- **`RunMetricsSnapshot` interface.** Nine readonly fields: `outcome` (`"completed" | "budget-stopped" | "aborted"`), `inputTokens`, `outputTokens`, `costUsd` (direct own-only, excluding nested sub-runs), `totalInputTokens`, `totalOutputTokens`, `totalCostUsd` (full subtree), `turns` (direct agent-turn events), `durationMs` (wall-clock). `replay()` and `replayStream()` do not fire the hook.
- **Double-fire semantics.** When a coordinator run dispatches sub-runs, each sub-run fires `onRunComplete` from its own execution context. The parent additionally fires `onSubRunComplete` when it sees the `sub-run-completed` event. This is intentional: use `onRunComplete` for aggregate metrics, `onSubRunComplete` for parent-perspective metrics.
- **Zero overhead when absent.** All hook branches are guarded by optional chaining (`options.metricsHook?.onRunComplete`). No counter allocation, no branch cost, no Map allocation when no hook is configured.
```
